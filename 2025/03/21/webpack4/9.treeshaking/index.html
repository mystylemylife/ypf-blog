9.treeshaking

发表于2025-03-21|更新于2025-03-25|webpack4

什么是treeshaking?

  • 中文即摇树,即摇掉多余的代码(死代码). 最初是由rollup实现,后来webpack自2.0开始实现.
  • treeshaking是基于ESModule进行静态分析导入导出, 确定模块中哪些导出值未曾被其它模块使用,然后删除掉
  • 不能处理commonjs动态执行的情况.

配置

  • webpack会在打包过程中,使用usedExports对要剔除的代码进行先标记,然后通过Terser插件进行后剔除
  • 实际测试,在开发模式下,只会看到部分代码标记,有些没用的代码没有被标记,但是生产模式剔除掉了.所以还是建议直接生产模式看

场景

我们可以看到

  • 默认全局js、css、直接执行语句都会被保留
  • 导入模块如果重新赋值但未使用也会被剔除
  • 未导出或者导出但未使用也会被剔除

默认webpack会认为当前项目有副作用, 我们可以通过在package.json中配置sideEffects来告诉webpack当前项目无副作用,从而来控制treeshaking.

sideEffects(副作用)

有副作用

测试结果就是上面的场景,说明webpack默认项目有副作用,但会自动剔除一些没用的代码.

无副作用

  • 全局js、css被剔除
  • 只有导入模块被使用和直接执行语句被保留
  • 其他未使用或重新赋值未使用被剔除

指定文件副作用

  • common.js会被保留, css被剔除
  • 未使用或重新赋值未使用被剔除
  • 使用的会被保留

在loader配置sideEffects

  • 此时所有css文件都默认有副作用, 会保留

babel-loader

我们知道treeshaking是基于ESModule进行静态分析的, 我们在用babel处理js的时候,Babel可以将 import/export 风格的 ESM 语句等价转译为 CommonJS 风格的模块化语句,所以要关掉.

  • 测试babel7版本,默认modules就是false,可以选 commonjs|cjs|amd|umd|false等
  • 只是在Babel编译环节编译禁止处理es，后面最终webpack转换仍为commonjs

#pure标注

  • 经/*#__PURE__*/标注的代码会被剔除

支持treeshaking的包

如果可以的话，应尽量使用支持 Tree Shaking 的 npm 包，例如：

  • 使用 lodash-es 替代 lodash ，或者使用 babel-plugin-lodash 实现类似效果

总结

  • package.json
    • 如果默认(有副作用),全局文件和使用到会保留,其他剔除
    • 如果设置无副作用, 那么只有使用到的会保留,其他包括全局文件会被剔除
    • 如果指定文件,那么只有指定文件和使用到的会保留,其他被剔除
  • loader中配置副作用, 文件会被保留
  • babel处理js,要关掉处理es
  • 经过/*#__PURE__*/标注的代码(同一行或下一行), 会被剔除
  • 尽量使用支持treeshaking的包,如lodash-es

原理

Webpack 在编译时会进行以下步骤：

  • 收集所有导出：分析每个模块的 export 语句，记录所有导出的变量/函数。 注意,必须是es模块(import/export，而非require)
  • 追踪导入并使用：从入口文件开始，记录哪些导出被import并分析哪些导入被实际使用（例如被调用、赋值等）。
  • 标记未使用的导出：未被任何模块导入或导入后未被实际使用的导出会被标记为 /* unused harmony export */。
  • 剔除未使用的导出：通过 Terser 插件对代码进行压缩，剔除标记为 /* unused harmony export */ 的导出。

那么如何判断import后是否被实际使用呢？

  • 直接调用（如 foo()）、属性访问（如 obj.key）、逻辑运算等会被视为使用。
  • 仅导入但未操作（如 import { foo } from ‘bar’ 后未调用 foo）会被标记为未使用。
    实际Webpack 在编译阶段会将代码解析为 抽象语法树（AST），然后遍历 AST 节点，分析变量的引用关系。具体判断逻辑如下：AST分析：
    Webpack 会识别 CallExpression 节点，并检查其调用的函数名是否匹配导入的变量（foo）。
    总之,就是会判断各种ast节点并匹配导入的变量来判断是否被使用.
