流程解析

  1. 初始化参数: 从配置文件和 Shell 语句中读取与合并参数，得出最终的参数。
  2. 初始化Compiler实例: let compiler = new Compiler(options.context)
  • 定义各种hook: 如entryOption、compile、make、compilation、normalModuleFactory、emit、done等
  • 定义run方法(编译的核心)

  1. 挂载plugin: 从配置参数对象,读取plugins数组循环订阅注册

  2. 注册并执行entryOption入口钩子: 回调中注册make钩子,从这里开始判断单/多入口进行编译以及入口的依赖编译

  3. 开始编译: 执行compiler.run((err, stats) => {console.log(err,stats.toJson())}). (下面都是run里面的代码,为了排版好看不缩进,从第6步重新开始, 不放到该步里面进行缩进)

  4. 创建compilation实例. 用于: 整个工程的编译模块汇总(上面模块工程是一个模块)

  • 定义各种属性
    • entries: 入口模块,只有入口
    • modules: 所有模块(比如入口模块依赖另一个模块, 就是入口模块、依赖模块)
    • chunks: 最终生成的所有chunk
    • assets: 最终生成的所有资源(根据chunks生成,内容差不多,区别是assets多了个通过ejs(webpack用的是自己字符串拼接的模版,这边方便用ejs)打包后的最终的代码,详见打包后的模块代码),用于遍历来生成dist中的一个个chunk文件
  • 定义createModule方法
    • 创建normalModuleFactory模块工厂实例. 用于: 编译模块(js解析为ast,递归找到require/import依赖模块存入dependencies数组)
      • 定义一个模块的各种属性:
        • 模块id(相对于根目录的路径)moduleId: 打包后自执行函数的入参模块对象key, 如 ./src/index.js
        • 模块编译后的代码_source: 比如require(xx)变为__webpack_require__(xx)
        • 模块ast_ast
        • 该模块依赖的模块dependencies: 是个数组,里面也是一个个模块对象包含上面这些属性
        • 该模块依赖的异步模块(import(xx))数组blocks
        • 该模块是否是异步模块标记 this.async = async
      • 定义build方法
        • 根据配置options找到rules,遍历判断如果和resource匹配,则通过loader-runner的runLoaders方法进行loader解析为js代码
        • 通过parser将loader解析的的js代码解析为ast
        • 分析判断ast节点是import/require, 说明是依赖同步模块,塞入dependencies数组中.同时require(xx)改为__webpack_require__(xx). 注意⚠️,塞入dependencies前有段逻辑
          • 依赖的同步模块需要判断路径是否startsWith("."), 是的话是本地模块, 不是的话需要把模块路径拼接上node_modules
        • 分析判断ast节点import如果是异步模块(import(xx)),说明是异步模块,塞入blocks数组中,塞入的数据标记async:true.同时import(xx)改为__webpack_require__.e(xx).then(__webpack_require__.t.bind(null,"${depModuleId}"))
        • 遍历blocks调用addEntry进行入口添加(因为动态模块相当于一个入口,需要单独生成chunk)
    • 将module实例塞入modules,并判断是入口,存入entries
    • 调用module.build编译,并将module.dependencies并塞入modules中
  • 定义seal方法,英文密封.用于: 遍历entries, 塞入chunks, 然后遍历chunks,同时调用mainRender模版塞入assets用于最终生成dist文件
  1. 执行make钩子触发从入口开始编译: compilation.addEntry(context, entry, "main", callback)里面调用了createModule方法
  2. 调用seal开始封装模块成(assets、moudles,chunks、entries )
  3. 触发emit钩子: 写插件用的最多, 因为此时开始写硬盘了, 这是修改的最后一步
  • 根据配置文件ouput属性创建dist目录
  • 遍历assets通过fs写文件
  1. 触发done钩子: 输出文件完成.

总结

webpack核心五大类

  • Compiler: 协调整个编译过程, 只在初始化执行一次,主要定义遍布整个编译生命周期的hooks以及run方法(创建Compilation实例、NormalModuleFactory实例)
  • Compilation: 描述一次编译过程, 每次改动文件编译都会创建一个新的Compilation实例, 就是代码编译的产物(assets、chunks、modules、entries)
  • NormalModuleFactory: 创建一个模块,用于编译模块: ast找依赖、require改__webpack_require__(js解析为ast,递归找到require/import依赖模块存入dependencies数组)、定义模块id(即相对于根目录的路径)用于打包后的自执行函数的模块入参key、require改__webpack_require__
  • Webpack: 解析参数,初始化Compiler实例, 注册入口钩子用于开始从入口开始编译
  • template: 提前定义好模版字符串, 遍历modules作为打包后的模版入参,拼接到模版中用于最终输出文件的内容(webpack是自己封装的模板, 这里为了简单,用的ejs),

简单流程

  1. 初始化参数: 从配置文件和 Shell 语句中读取与合并参数，得出最终的参数。
  2. 初始化Compiler实例: 定义遍布整个编译生命周期的hooks和run方法
  3. 调用compiler.run((err, stats) => {console.log(err,stats.toJson())})方法开始执行编译
  4. 定义entryOption入口钩子,从入口开始进行编译, 调用compilation.createModule创建模块, 该方法调用const module=new ModuleFactory()创建一个模块,包括
  • moduleId-打包后自执行函数的入参模块对象key, 如 ./src/index.js
  • 读取webpack配置的rules遍历,正则判断文件路径匹配到的话,调用loader-runner的runLoaders方法进行匹配到的loader解析为js代码,然后将js通过parse解析为ast
  • 分析ast, 该模块的同步依赖模块存入dependencies同时require改为__webpack_require__后,再ast转源码的_source
    • 依赖的同步模块需要判断路径是否startsWith("."), 是的话是本地模块, 不是的话需要把模块路径修改拼接上node_modules
  • 分析ast, 该模块的异步依赖模块存入blocks同时import(xx)改为__webpack_require__.e(xx).then(webpack_require.t.bind(null,”${depModuleId}”))后,再ast转源码的_source
  • 遍历blocks调用addEntry方法进入入口添加异步模块(异步模块实际就相当于一个入口,需要生成单独的chunk)
  1. 构建和封装compilation模块产物: assets、moudles,chunks、entries
  • 将上面定义的模块module存入modules, 入口模块通过addEntry存入entries, 遍历module.dependencies塞入modules.
  • 遍历entries, 塞入chunks,同时调用mainRender模版(提前定义好了输出文件的结构)生成打包后的文件内容字符串塞入assets用于最终生成dist文件
  1. 触发emit钩子,写插件用的最多, 因为此时开始写硬盘了, 这是修改的最后一步
  2. 根据webpack.config.js配置文件的ouput属性创建dist目录
  3. 遍历assets通过fs写文件
  4. 触发done钩子表明输出文件完成.
