HTTP缓存

发表于2023-12-25|更新于2025-06-19|http

本片文章从2个方面理解http缓存

1.缓存的分类详解 2.nginx缓存实战

缓存的作用

  • 节省带宽
  • 更快的加载页面
  • 减少服务器负载，避免服务器过载的情况出现。

缓存的分类

http缓存分为强缓存和协商缓存

强缓存

1.强缓存使用Expires、Cache-Control 两个头字段。
2.当命中强缓存的时候，客户端不会再发请求，直接从缓存中读取内容，并返回HTTP状态码200。

Expires

具体流程如下：
当浏览器请求服务器的时候，服务器会返回资源的同时还会带上响应头Expires，表示资源的过期具体时间，设置这个字段之后，就可以告诉浏览器，在未过期之前不需要再次请求，可以直接在缓存里面拿。

上面例子表示缓存在2023-01-30 14:25:20过期，我们在js中可以通过 new Date().toGMTString() 来获取GMT时间

1.它是HTTP/1.0版本中定义缓存的字段
2.它表示缓存过期的绝对时间，是一个格林尼治(GMT)格式的标准时间。
3.由服务端设置，所以它只会出现在响应头中

但其存在一个缺点，

  • 由于是绝对时间，用户可能会将客户端本地的时间进行修改，而导致浏览器判断缓存失效。此外，即使不考虑用户修改，客户端和服务端时间存在时差或者误差等因素也可能造成客户端与服务端的时间不一致，致使缓存失效。所以http/1.1 版本新增了Cache-Control字段。Expires和Cache-Control同时存在，Cache-Control优先级高

Cache-Control

1.它是HTTP/1.1版本中定义缓存的字段
2.它可以表示一个相对时间也可以设置其他值
3.一般由服务度设置，客户端可以设置，所以它既能出现在请求头又能出现在响应头，代表不同意思

下面是Cache-Control一些常用的属性值：

  • max-age: 决定客户端资源被缓存多久，单位为 s, 是一个相对时间。当为0的时候就是不使用强缓存，类似no-cache。
  • no-cache: 表示不使用本地强缓存，需要使用协商缓存。
  • no-store: 表示不缓存，永远从服务端拿
  • public: 表示资源可以被所有的用户缓存，包括终端用户即浏览器和中间代理服务器。
  • private: 只能被终端用户的浏览器缓存，不允许中间缓存代理进行缓存。（默认值）
  • s-maxage: 用于表示 cache 服务器上（比如 cache CDN，缓存代理服务器）的缓存的有效时间的，设置时候需要同时设置public，如 Cache-Control: max-age:3600, s-maxage=3600, public

注意： Cache-Control的属性值，它们可以组合使用，用逗号隔开，但是no-cache和no-store是一组互斥属性，不能同时使用
比如设置了Cache-Control:max-age=1800,即30分钟缓存. 当12:00访问,此时响应头也会有Expires字段,显示的是Thu, 20 Feb 2025 12:30:00 GMT的过期时间。在这中间我们访问都是强缓存, 如果12:31访问,会重新变成304协商缓存看是否更新, 然后再强缓存.

协商缓存

在强缓存失效（过期等原因，不是不存在）后，客户端会携带缓存标识向服务端发起请求，服务端会根据缓存标识决定是否使用缓存。

协商缓存的标识字段有两组，Last-Modified / If-Modified-Since和Etag / If-None-Match，如果同时存在，Etag / If-None-Match的优先级比Last-Modified / If-Modified-Since高。
由服务端设置，下次请求时浏览器携带过去
协商缓存生效时，状态码为304，返回的仅仅是一个状态码而已，并没有实际的文件内容，所以请求时间和资源的大小都会极大的减少

Last-Modified / If-Modified-Since

当客户端第一次请求服务器的时候，服务端会返回一个Last-Modified响应头，该字段是一个GMT标准时间。下次客户端请求服务器的时候会带上If-Modified-Since请求头字段，该字段的值就是服务器上次返回的Last-Modified的值。服务器接收到请求后会比较这两个值是否一样，一样就返回304，让客户端从缓存中读取，不一样就会返回状态码200，并返回新文件给客户端并更新Last-Modified响应头字段的值。

但是存在一个缺点：

  • Last-Modified使用GMT只能精确到秒，如果一个文件在1s内进行修改，客户端在1s内进行多次请求，获取到的仍是缓存
  • 如果只单纯编辑了下文件，如编辑又撤销，并没有更新内容，Last-Modified会被更新，再次请求服务端对比时间，导致缓存失效

所以为了解决上面问题，又引入了Etag 和 If-None-Match

Etag 和 If-None-Match

当客户端第一次请求服务器的时候，服务端会返回一个Etag响应头。客户端请求服务器的时候会带上If-None-Match请求头字段，该字段的值就是服务器返回的Etag的值。服务器接收到请求后会比较这两个值是否一样，一样就返回304，让客户端从缓存中读取，不一样就会返回新文件给客户端并更新Etag响应头字段的值。

Etag 是由文件内容的 hash 值和内容长度计算而成的唯一哈希值，只有当文件文件内容发生变化了Etag的值才会发生变化。

虽然etag解决了上面1s内改动以及文件编辑未更新的问题，但是etag也有缺点

  • ETag需要计算文件指纹这样意味着，服务端需要更多的计算开销。如果文件尺寸大，数量多，并且计算频繁，那么ETag的计算就会影响服务器的性能。显然，ETag在这样的场景下就不是很适合。
  • ETag有强验证和弱验证，所谓将强验证，ETag生成的哈希码深入到每个字节。哪怕文件中只有一个字节改变了，也会生成不同的哈希值，它可以保证文件内容绝对的不变。但是，强验证非常消耗计算量。ETag还有一个弱验证，弱验证是提取文件的部分属性来生成哈希值。因为不必精确到每个字节，所以他的整体速度会比强验证快，但是准确率不高。会降低协商缓存的有效性。

所以，不同于cache-control是expires的完全替代方案。ETag并不是用于last-modified的替代方案。而是last-modified的补充方案，项目中到底是用ETag还是last-modified完全取决于业务场景，这两个没有谁更好谁更坏

启发式缓存

上面我们介绍了强缓存和协商缓存,实际浏览器还有一种默认缓存方式: 启发式缓存.
不知道大家有没有遇到过这种情况，就是服务器的响应头里没有返回 Expires 和 cache-control。而是返回了 Last-Modified 和 ETag.理论上这种情况下浏览器应该要走协商缓存，但是结果却返回的 200（from memory cache），直接走的强缓存.这就是启发式缓存。

缓存到底存在哪？

上面我们讲了缓存，那么缓存到底存在哪？浏览器？电脑硬盘？
实际上，我们可以在浏览器控制台看到缓存的存储方式。

  • memory cache: 内存中的缓存，按照操作系统的常理：先读内存，再读硬盘。
  • disk cache: 硬盘上的缓存
  • Service Worker: js中的service worker，因为上面2种缓存方式是浏览器自己的行为，浏览器自己控制从内存还是硬盘读取，前端无法控制，service worker给予了我们另外一种更加灵活，可以直接的操作方式，需要手动执行cache.delete(resource)进行清除

  • 我们可以看到，第一次访问，两条请求的状态码都是 200。html和静态资源的响应头都会有 ETag和Last-Modified，这不就是协商缓存的字段嘛， 所以nginx默认开启了缓存。
  • 我们刷新页面，发现html的状态码是304，其他静态资源是200，来源于缓存，也就是说每一次html文件都是命中了协商缓存，而js文件都是命中了强缓存(启发式缓存)。

细心发现，我可以看到，刷新页面后，

  • html的请求头里面由原来的Cache-Control: no-cache变成 Cache-Control: max-age=0并且带上了 If-Modified-Since: Tue, 17 Jan 2023 07:17:36 GMT和 If-None-Match: "63c64b90-31d", 可以看到此时浏览器携带上了协商缓存的相关参数，以此来避免命中强缓存。
  • 静态资源的请求头如下图是启发式缓存,所以是强缓存（走的缓存看不到具体header信息，浏览器隐藏了，可以抓包看）

现在我们修改配置

no-cache

结果如下：

  • html以及静态资源响应头都增加了Cache-Control: no-cache，之前没配置nginx，都没有。
  • 静态资源都是协商缓存304,请求时候请求头都带上了If-Modified-Since和If-None-Match
  • html和没配置一样，请求时候请求头仍是Cache-Control: max-age=0并且带上了 If-Modified-Since和If-None-Match

no-store

结果如下：

  • html以及静态资源响应头都增加了Cache-Control: no-store
  • html和静态资源不走缓存，永远取最新的
  • 静态资源请求头没有了If-Modified-Since和If-None-Match和Cache-Control
  • html请求头没有If-Modified-Since和If-None-Match，但有Cache-Control: max-age=0

max-age=5

我们配置缓存5s,结果如下

  • html以及静态资源响应头都增加了Cache-Control: max-age=5
  • html永远是304(请求时永远都携带Cache-Control: max-age=0以及 If-Modified-Since和If-None-Match)，
  • 静态资源在5s内一直走强缓存，当失效时， 再次刷新页面， 静态资源带上If-Modified-Since和If-None-Match发起协商缓存304请求，再次刷新页面，静态资源重新变成强缓存

综合上面三种测试，我们发现

  • 上面三种无论怎么设置，nginx永远对html进行协商缓存，静态资源根据不同配置，不同结果

缓存场景

  • 我们现在都是工程化打包，拿vue举例子，打包后的静态资源都有hash值，所以说走强缓存即可
  • html走协商缓存或者不缓存，html因为引用静态资源，又没有hash值，所以更新，都要新的
  • 对于不常变化的静态资源，我们可以单独设置一个很长的max-age，比如max-age=31536000 (一年),这样1年内走强缓存，连协商缓存的请求都省去了。但是为了解决更新问题，需要在url上加版本号/hash等来控制

浏览器缓存

最后， 前端中还有个浏览器缓存，简单描述下

  • cookie
  • LocalStorage/SessionStorage
  • indexedDB
  1. cookie大小为4KB，每个域名下的cookie数量最多为20个（但很多浏览器厂商在具体实现时支持大于20个）
  2. cookie的key和value如果里面含有;号，设置都时候就会被截断，因为我们通过document.cookie设置本来就是字符串拼接并分号隔开的，所以需要编码下， 实际上中文等特殊字符我们都最好编码下
  3. localStorage/sessionStorage大小为5M，我们通过localStorage.setItem(key,value)设置的，所以不会存在截断问题，但是特殊字符啥的最好编码下
  4. IndexedDB为解决前端无法存储大容量数据而生，它的存储容量非常大，取决与你的硬盘。相当于前端的数据库。主流浏览器都支持，看兼容性，可以上caniuse查询下
