前端安全-XSS攻击

发表于2023-12-25|更新于2023-12-25|javascript

XSS攻击是什么?

XSS定义

XSS攻击,中文名叫跨站脚本(Cross Site Script)攻击,缩写应该是 CSS，但是为了和 CSS(Cascading Style Sheet)层叠样式有所区别，所在在安全领域叫做 “XSS”。

XSS分类

  • 反射型XSS（Reflected XSS）
  • 存储型XSS（Stored XSS）
  • DOM-based XSS

下面我们来一一介绍

反射型XSS（Reflected XSS）

顾名思义，反射型 XSS，只是将用户输入的数据“反射”给浏览器。其是一种非持久型xss攻击，依赖于服务器对恶意请求的反射,仅对当次的页面访问产生影响
常见于通过 URL 传递参数的功能，如网站搜索、跳转等场景。

攻击步骤如下:

  1. 攻击者构造含有script脚本的恶意链接,诱导用户点击
  2. 用户点击后,服务端取出url上的恶意代码,直接返回前端(前端通过innerHTML直接插入代码),或者以前的jsp模式,后端直接拼接html返回渲染,就会触发反射型XSS

举个🌰:
比如访问链接:https://test.com/index.html?q=<script>alert(1)</script>,页面里面含有从服务端取出content字段,并且document.getElementById("seachKeyWords").innerHTML = content,用来展示当前搜索的内容的逻辑,这样就会触发反射.

存储型 XSS（Stored XSS）

顾名思义，存储型 XSS，就是攻击者的数据已经存储在服务端，比如用户在文本框中输入评论，然后提交保存到后台。然后其他用户浏览评论，导致触发存储型 XSS。其是一种持久型xss,攻击时效将伴随着攻击数据一直存在。
常见于带有用户保存数据的网站功能，比如论坛发帖、商品评价、用户私信等场景。

攻击步骤如下:

  1. 攻击者将恶意代码作为参数提交到目标网站的数据库中
  2. 用户访问页面,页面渲染接口返回的数据触发xss,如果多用户可以访问,都会触发

举个🌰:
比如用户评论某篇文章, 评论内容是<script>alert(1)</script>,提交到后台,前端通过innerHTML直接渲染到评论区,那么所有用户访问该评论区,都会alert,所以存储型 XSS一旦被发现,影响巨大

DOM-based XSS

从效果上看,DOM-based XSS其实也属于反射型XSS,但是其区别是, 反射型的xss脚本来自于服务端(如前端提交到后台,后台处理后返回), dom型的xss脚本来自于前端,和服务端没关系,dom型属于前端自身的漏洞,和后端没关系.
常见于从URL参数、cookie、LocalStorage中取内容的场景。

攻击步骤如下:

  1. 前端 JavaScript 取出 URL中的恶意代码
  2. innerHTML插入代码导致渲染触发xss

举个🌰:
比如访问链接:https://test.com/index.html?q=<script>alert(1)</script>,页面里面含有直接从url上取出content字段,并且document.getElementById("seachKeyWords").innerHTML = content,用来展示当前搜索的内容的逻辑,这样就会触发 DOM-based XSS

如何触发、危害?

如何触发

  1. 反射型和dom型,难点在于构造恶意链接如何让用户去点击,所以常见的方式是通过邮箱、短信等发送给用户,诱导用户点击
  2. 存储型依靠代码漏洞,通过提交恶意数据到数据库来触发

危害

  1. 窃取用户信息进行违法行为
  2. 引流(嵌入重定向脚本)

常见XSS代码方式?

谷歌内置拦截器会拦截background:url标签

总结下:

  1. 避免html内联事件绕过:如 onclick、onerror、onload、onmouseover 等,在 JavaScript 中通过 .addEventlistener() 事件绑定会更安全
  2. 在插入html属性要做过滤处理
  3. a标签的href、img的src、background的url
  4. svg图片含有js代码
  5. JavaScript 的 eval()、setTimeout()、setInterval()、location.href、windowopen 等都能把字符串作为代码运行

注意: 谨慎使用.innerHTML、.outerHTML、document.write() ，不要把不可信的数据作为 HTML 插到页面上。如果使用的模版,比如ejs,尽量使用 <%= data %> 而不是 <%- data %>； 如果使用框架(vue),尽量使用{{}},少使用v-html ;

如何防御?

上面我们讲了常见的xss代码攻击方式,那么我们如何防御呢?
1.转义字符
对于用户的输入应该是永远不信任的，最普遍的做法就是转义输入输出的内容，对于括号，尖括号，斜杠进行转义

实际上,我们需要根据具体场景使用不同的转义规则
常用的前端插件有 xss.js，DOMPurify

2.设置Cookie httpOnly
禁止 JavaScript 读取某些敏感 Cookie，攻击者完成 XSS 注入后也无法document.cookie获取cookie

3.输入内容长度、类型的控制
对于不受信任的输入，都应该限定一个合理的长度，并且对输入内容的合法性进行校验（例如输入email的文本框只允许输入格式正确的email，输入手机号码的文本框只允许填入数字且格式需要正确）。虽然无法完全防止 XSS 发生，但可以增加 XSS 攻击的难度。

4.CSP
内容安全策略(Content-Security-Policy)

vue XSS?

我们在使用vue中,常用的插入内容标签有我们找到双括号源码,发现,vue是通过createTextNode生成文本字符串,所以不会存在xss, 不过v-html是有风险的,需要谨慎使用 .
