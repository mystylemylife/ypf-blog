4.loader

发表于2024-12-30|更新于2025-02-12|webpack4

为什么会有loader

  • webpack只能处理.js和.json文件,但打包过程中遇到其他类型文件，如.vue、.ts、图片、.css等，webpack就无能为力了,面对这一问题，webpack提供了loader机制,用来将webpack不能识别的文件，转换为标准的js模块.
  • loader的本质就是一个函数,webpack内部的loader runner会调用此函数，然后将上一个loader产生的结果或者资源文件传入进去

用法

我们通常使用loader是在配置文件中配置的，除去inline-loader下面会讲到

我们看到, 使用的时候有多种写法

  • 我们可以use: 对象、字符串、数组、数组对象, 一般我们用最后一种,可以传多个且能传options进行配置
  • loader可以引用本地路径,也可以引用node_modules中的loader
  • 如果觉得本地路径写法不优雅,我们通过resolveLoader配置webpack查找loader方式

loader分类

在webpack中,从loader的执行时机, 可以分为4种,执行顺序: pre -> normal -> inline -> post

  • pre: 前置loader
  • normal: 普通loader
  • inline: 内联loader
  • post: 后置loader

配置loader

  • pre和post和normal都是通过rule对象的enforce配置的, 不传enforce, 默认是normal

内联loader

  • inline比较特殊,它只能在代码中引入模块的时候进行声明, 语法如下,官方文档loader
  • 使用!进行分割, 最后一块是要解析的文件
  • 每个loader都可以配置options, 类似query参数
  • 我们可以在最前面加上3种方式来控制配置loader如何执行
    • !: 不执行normal-loader
    • !!:不执行所有配置loader
    • -!: 不执行pre-loader和normal-loader

异步loader

上面四种loader默认都是同步的,如果需要异步, 我们可以这样

  • 调用async方法,可以把loader执行从同步改为异步
  • 虽然loader此时是异步的,但是此时会进入阻塞状态, 只有等待调用callback后,才会执行下一个loader,比如 loader1、异步loader2、loader3, 执行顺序是: 先输出loader1、5s后输出loader2、最后输出loader3

loader执行顺序

无pitch

  • 从种类来看: pre -> normal -> inline -> post
  • 从执行方向看: 倒序执行/从右向左/从下往上, 比如use:[loader1 ,loader2 ,loader3], 执行顺序是: loader3 -> loader2 -> loader1

  • 实际loader的执行是调用了loader-runner的runLoaders方法.

loader-runner

我们需要了解下loader-runner的运行机制. 假如我们有一个loader的配置如下: !loader1!loader2!loader3!./src/index.js

  • 先切割变成数组[loader1,loader2,loader3]传递给loader-runner
  • loader-runner会构造一个上下文loaderContext对象,供执行pitch和normal函数的this指针
    • loaders: 上面loader数组转换为数组对象,每个loader对象包含
      • loader的全路径path
      • query参数
      • pitchExecuted: 是否执行了pitch方法, executed:英文执行的意思
    • request: 将loaders全路径数组转换为内联字符串方式: xx/loader1!xx/loader2!xx/loader3!D:/src/index.js
    • resource: 要被处理的的resource,如D:/src/index.js
    • loaderIndex: 当前执行loader函数(pitch/normal)的索引
    • data: 当前执行loader的data参数
    • currentRequest: 当前loader+剩余loader+resouce字符串
    • remainingRequest: 剩余loader+resouce字符串, 不包括当前loader的request.
    • previousRequest: 之前loader字符串, 不包括当前loader的request.
  • 循环loaders, 依次执行loader的pitch函数,loaderIndex++
    • loaderIndex=0: 此时开始执行loader1-pitch
      • currentRequest:xx/loader1!xx/loader2!xx/loader3!D:/src/index.js
      • remainingRequest:xx/loader2!xx/loader3!D:/src/index.js
      • previousRequest:''
    • loaderIndex=1: loader1-pitch执行完毕, 执行loader2-pitch
      • currentRequest:xx/loader2!xx/loader3!D:/src/index.js
      • remainingRequest:xx/loader3!D:/src/index.js
      • previousRequest:xx/loader1
    • loaderIndex=2: loader2-pitch执行完毕, 执行loader3-pitch
      • currentRequest:xx/loader3!D:/src/index.js
      • remainingRequest:D:/src/index.js
      • previousRequest:xx/loader1!xx/loader2
  • 依次执行loader的normal函数,loaderIndex--
    • loaderIndex=2: loader3-pitch执行完毕,开始执行loader3-normal
      • currentRequest:xx/loader3!D:/src/index.js
      • remainingRequest:D:/src/index.js
      • previousRequest:xx/loader1!xx/loader2
    • loaderIndex=1: loader3-normal执行完毕,开始执行loader2-normal
      • currentRequest:xx/loader2!xx/loader3!D:/src/index.js
      • remainingRequest:xx/loader3!D:/src/index.js
      • previousRequest:xx/loader1
    • loaderIndex=0: loader2-normal执行完毕,开始执行loader1-normal
      • currentRequest:xx/loader1!xx/loader2!xx/loader3!D:/src/index.js
      • remainingRequest:xx/loader2!xx/loader3!D:/src/index.js
      • previousRequest:''
  1. 通过上面流程我们可以知道pitch和normal函数的三个字段输出是一样的, 我们可以按pitch来理解这三个字段的结果
  2. remainingRequest这个属性比较重要,pitch函数就需要用到这个属性来将剩余的loader变成内链执行.(因为pitch跳出就没法执行后续了,所以在这里手动执行)

url-loader

url-loader可以将打包过程中用到的文件转换为 base64 格式字符串，这样可以减少网络请求次数。比如100个小图片,处理后,只有1个文件, 减少100次请求, 优化了网络请求。但注意需要控制阈值,否则会导致文件过大

总结

  • 判断阈值以及获取文件类型插入base64字符串
  • 如果不满足阈值，则使用file-loader, 所以我们如果用了urrl-loader，不用再在webpack.config.js中配置file-loader了, 但需要安装, 因为会依赖

style-loader和less-loader

style-loader 可以将 css 代码注入到 js 中，运行时，通过创建 style 标签，将样式插入到 head 中。
less-loader 将 less 转为 css, 然后交给 style-loader 处理。 sass-loader同理

css-loader

总结

  • css-loader依赖postcss解析ast
  • 分析ast,找到所有@import xx规则的url, 转换为相对路径并拼接上css-loader并require, 通过内链的方式进行执行
  • 分析ast,找到所有background:url(xx)规则, 转换为相对路径并require,供webpack的其他loader解析
  • 因为都要通过require引用, 所以肯定是个js脚本, 所以需要style-loader在pitch就得执行,而不是css-loader执行导致style标签中的是js脚本

简单来说，如果在loader开发中你的需要依赖其他loader，但此时上一个loader的normal函数返回的并不是处理后的资源文件内容而是一段js脚本，那么将你的loader逻辑设计在pitch阶段无疑是一种更好的方式。

babel-loader

我们来实现一个babel-loader来进行es6转es5,希望下面箭头函数转换为es5

总结

  • 原理很简单, loader本质上就是一个函数, 最终通过this.callback给webpack返回结果, webpack会根据返回结果进行下一步操作
    • callback接收三个参数: code-转换后的es5代码, map-sourceMap文件, ast-抽象语法树
    • 因为babel转换后提供了ast, 所以调用callback传进去, 那么webpack会直接使用这个ast, 不会自己再通过acron解析器将code转ast了, 提高效率, 所以如果loader有ast,可以直接用
  • loader的sourceMap和webpack.config.js中的devtool:’source-map’需要同时开启,才会生效
    • webpack的devtool:source-map、loader的sourceMap:true, 谷歌控制台调试会看到index.js, 是es6源代码
    • webpack的devtool:source-map、loader的sourceMap:false, 会看到index.js, 是es5代码
    • webpack的devtool:false、loader的sourceMap:true, 会看到main.js, 是es5代码
    • webpack的devtool:false、loader的sourceMap:false, 会看到main.js, 是es5代码
