import按需加载

我们知道,webpack打包默认把所有模块都会打包出一个js, 如上节讲的 main.js.这样会导致文件过大,所以我们需要按需加载(分割js)

打包运行如下图:


上图, title被打包到1.js, title-es被打包到0.js. 剩下就是主入口main.js

  • 在webpack中, 只要通过import('xxx')引入的模块,打包的时候都会生成单独的js文件,不会合并到入口js中.
    • 这里说的是import('xx')引入文件,会生成单独js
    • 不是import xx from 'xxx'这种引入属性,这种还是合并到入口js中
  • 如果通过import引入模块里面内部也有通过import引入, 其内部引入的也会生成单独的js, 所以只要import引入模块都会生成单独js
  • 默认分离的模块名是webpack自己生成的, 当然也可以使用webpack提供的魔法注释自定义
  • 返回是一个promise,查看then返回结果,无论es还是commonjs模块, 都会变成下面这种es对象
    • default: 模块的默认导出
    • keys: 模块的属性
    • __esModule: true
    • Symbol(Symbol.toStringTag): Module

魔法注释

魔术注释(Magic Comments)是由 Webpack 提供的，可以为代码分割服务的一种技术。通过在 import 关键字后的括号中使用指定注释，我们可以对代码分割后的 chunk 有更多的控制权。

  • webpackChunkName: 用来指定打包后的文件名
  • webpackPrefetch: 资源预拉取
  • webpackPreload: 资源预加载
  • …等等

webpack_require.e 动态加载chunk

核心方法: 用来动态加载模块chunk

demo代码__webpack_require__.e(1)逻辑梳理如下:

  • 全局变量installedChunks 用来缓存模块加载状态, 初始化默认 main主入口模块,状态为0表示加载成功
    • undefined: 未加载
    • null: 预加载/预获取
    • Promise: 正在加载中
    • 0: 加载成功
  • 判断installedChunks[chunkId]是否为0, 第一次肯定不为0,是undefined
    • 创建一个promise, installedChunks[chunkId]=[resolve, reject, promise],同时定义promise数组存入promise.
    • document.createElement创建script,然后插入head标签中开始加载chunkId对应的模块. 同时开始120s的定时器超时.
    • 最后 return Promise.all(promises)

我们加载了chunk,那加载成功后,如后回调?

window.webpackJsonp桥回调函数

webpack是在window上定义了一个桥(window.webpackJsonp)用来存放执行加载成功的回调.

  • window["webpackJsonp"]是一个回调函数桥, 接收模块id和模块对象, chunkIds是模块id, moreModules是模块对象
  • 该桥定义push方法,让chunk来调用传入.(因为各个chunk都是独立文件,通过window定义变量操作就方便了)
    结合下面chunk调用push,我们理下逻辑:
  • 标记模块加载成功: installedChunks[chunkId] = 0
  • 将模块对象合并到我们上节课讲的自执行函数的modules参数中,因为初始化入参只包含了入口模块对象(非动态加载模块对象), 合并后就包含了各个chunk对象了
  • 循环执行resolve方法

webpack_require.t统一返回chunk对象

主要目的是将commonjs转换为es模块(增加了default属性), 因为es模块是有default属性, 为了和他一直,这样通过import的then拿到的格式一样.

  • __webpack_require__.t方法接收两个参数, 第一个是模块id, 第二个是模式, 模式是二进制位, 用来判断是否需要转换模块
  • 对于common模块主要是创建空对象,然后default保存整个对象, 每个属性合并过去, 增加__esModule: true Symbol(Symbol.toStringTag): Module,最后return这个对象
